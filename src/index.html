AI workflow system / skill operations

把 Agent Workflow
變成可維護系統

核心方法不是累積 prompt 技巧，而是把 agent workflow 拆成一套 可操作的 runtime system：human 和 agent 在同一個工作環境裡使用同一套 contracts， 由 skills、hooks、policies、CLI contracts、work tier 和 heuristic system 形成可驗證、可交接、可改進的工程流程。

Method Baseline

核心方法：先把工作流設計成 system

起點不是多寫幾個好用 prompt，也不是先整理一大包 skills。 先把 workflow 拆成可維護的層次：判斷留在 policies 和 skills， 穩定操作收斂成 CLI contracts，關鍵邊界交給 hooks， 執行結果留下 evidence，反覆出現的 friction 進入 heuristic loop。

方法前提

先設計環境，再寫 skill

Skill 不是獨立存在的 prompt 檔。它必須被放進明確的 repo policy、 validation rule、CLI primitive 和交付流程裡，才會變成可重用能力。

抽象原則

分開 judgment 和 deterministic primitives

Policies / skills 負責判斷與流程選擇；nils-cli / forge-cli 負責穩定輸入輸出、dry-run、provider-neutral behavior 和 JSON contract。

落地檢查

每個 workflow 都要能被驗證和改進

一個 workflow 是否成熟，取決於它是否有清楚的 trigger、stop condition、 validation、evidence path，以及失敗後能被壓縮成更好規則的機制。

Implementation Path

實作主線：從問題到 deterministic primitives

順序不是工具 demo，而是建立系統的方法：先界定不穩定來源， 再切出核心分層，接著加入 agent 自主判斷與學習迴路， 最後用實際 repo 證明每個抽象層都有可操作的落點。

Step 1：界定為什麼需要 agent workflow system

直接讓 agent 使用 shell、`gh`、`glab` 和散落資料，短期快，長期難驗證、難交接、難重複。

Step 2：定義核心模型

Policies 決定邊界，skills 決定 workflow，hooks 擋住危險路徑，CLI contracts 提供穩定工具面，frontends 讓不同場景重用同一套能力。

Step 3：加入 agent 自主判斷層

Work Tier 讓 agent 自動判斷任務該用多少 tracking；Heuristic System 讓 agent 把失敗和反覆學到的規則沉澱成可維護的系統。

Step 4：把案例對應回系統層次

forge-cli、nils-alfredworkflow、symphony-board：同一個核心工具能力，讓 human 和 agent 在 CLI、快速入口、dashboard 中使用。

Step 5：用最小閉環導入

從一個常見 workflow 開始：定義 skill contract、補 CLI/validation、接 hooks、保留 evidence，最後壓縮成可重用 playbook。

Problem

問題不是 agent 不夠強，是工作環境沒有被設計

一次性使用 agent 很容易靠 tacit knowledge 補洞；當 workflow 要被重複執行時， 每個洞都會變成不穩定行為。第一個動作，是把「臨場習慣」 轉成「可被 agent 和人共同執行的環境」。

Before

直接把工具丟給 agent

  • agent 直接呼叫 `gh` / `glab`，provider 行為差異散在 prompt 裡。
  • 任務大小、是否要開 issue、是否需要 plan 都靠當下判斷。
  • 錯誤或 workaround 留在聊天紀錄，下一次不一定會被記得。
  • 不同人用不同入口，CLI、UI、Alfred、agent 各自漂移。
After

把環境設計成 workflow system

  • CLI 提供 stable contract，skills 只描述 workflow decision。
  • Work Tier 讓 agent 在開始工作前主動判斷 tracking level。
  • Heuristic System 把重要 friction 轉成 tests、policy、runbook 或 retained records。
  • 人和 agent 使用同一套工具能力，只是入口不同。
核心句： 不只是讓 agent 會用工具，而是把人和 agent 放進同一個可驗證、 可約束、可交接、會持續改進的工作環境。

Core Model

核心模型：從意圖到工具，再回到學習

主軸不是每個 repo 的功能，而是這個分層：哪些東西是 judgment，哪些東西是 deterministic primitives，哪些東西只是不同場景的入口。

Policies 定義不可踩的邊界：Git、validation、external facts、docs placement、work tier。
Skills 把工作拆成可交接的 workflow contract：何時使用、前置條件、停止條件、驗證。
Hooks 把重要規則變成執行時 guardrail：preflight、direct command block、finish-line gate。
CLI Contracts 用 nils-cli / forge-cli 等工具提供 stable JSON、dry-run、provider-neutral behavior。
Frontends + Evidence CLI、Alfred、symphony-board、GitHub issue/PR、retained records 都是不同場景的可見入口。

Source of truth

agent-runtime-kit 是 shared runtime source：skills、hooks、policies、targets、manifests。

Primitive layer

nils-cli 承接 deterministic command contracts，避免 skill prose 自己處理 fragile mechanics。

Consumer surfaces

Core Practices

三個核心做法

01

Skills 不是 prompt snippet

Skill 應該是工作單元：它要說明何時用、前置條件、輸入輸出、 failure modes、驗證和 handoff。好的 skill 會降低判斷成本， 而不是把所有經驗塞成長文。

02

自動化之前要先有工具合約

agent 能呼叫 CLI 不代表工作流可靠。先把 provider 差異、dry-run、 JSON envelope、label validation、merge gate 收斂成 CLI contract， 再讓 skills 使用。

03

工作流要會自己學習

Work Tier 讓 agent 自主選 tracking level；Heuristic System 讓 agent 把重要失敗變成 retained learning。這是從「一次任務」 走向「長期系統」的關鍵。

Autonomous Decision Layer

Work Tier：讓 agent 自己判斷要多少 tracking

Work Tier 是這套工作流中的自主判斷層。它不是工具命令， 而是 agent 在開始工作前應該自主做的判斷：這件事是直接做， 還是需要 issue、plan tracking、甚至 dispatch lanes？

Repo evidence

來源： core/policies/work-tier-levels.md

設計重點

  • Tier 衡量的是 tracking machinery，不是程式碼大小。
  • PR 是 delivery floor；tier 是 state tracking level。
  • agent 要從 L0 開始，只有具體 trigger 才升級。
  • L1+ 會建立 durable provider artifact，所以要把決策顯性化。
agent 啟動判斷模板 
L0：這是直接交付的修改，PR 足以承載記錄，我會直接做並驗證。
L1：這需要 durable timeline 或 root-cause investigation，建議先開 follow-up issue。
L2：這是多步驟且需要 frozen plan / execution state 的工作，建議開 plan tracking issue。
L3：這需要多個獨立 lanes 或 subagents，建議走 dispatch plan。

Learning Loop

Heuristic System：讓工作流從失敗中變好

如果 Work Tier 是任務開始前的自主判斷，Heuristic System 就是任務結束後的 自主學習。它不訓練模型權重，而是把具體 workflow failure 轉成更好的 skill policy、validation、script、test、runbook 或 retained operation record。

要避免的誤解

  • 不是把所有錯誤都寫進 memory。
  • 不是把 raw logs 或暫時狀態 commit 進 repo。
  • 不是每個失敗都值得保留；要先 triage。
  • 真正有價值的是把重複 friction 壓縮成更小的規則或測試。

Repo evidence

來源： core/policies/heuristic-system/HEURISTIC_SYSTEM.md

1. Workflow result 任務出現 failure、retry、錯誤文件、模糊輸出或 repeated workaround。
2. Triage 判斷 ignore / summarize / retain，不把暫時錯誤誤升級成永久規則。
3. Durable form 可能是 error-inbox case、operation record、test、script、runbook、policy update。
4. Compression 當例外越來越多，要壓縮成更小、更穩定、更容易驗證的 contract。

Skills

Skills：把 tacit workflow 變成 explicit contract

Skill 的價值不是「幫 agent 多背一段指令」。 它應該把工作判斷切成可交接、可驗證、可演進的 workflow contract。

Repo evidence

來源： core/skills 與 core/skills/README.md

Skill contract checklist

  • 明確 trigger：何時該用，何時不該用。
  • 明確 boundary：skill 擁有什麼，不擁有什麼。
  • 明確 prerequisites：CLI floor、auth、branch state、docs、validation。
  • 明確 stop condition：什麼狀態下不要硬做。
  • 明確 validation：最小但足夠證明 workflow shape 的檢查。
實作順序：先不要問「要多少個 skills」。先選一個重複發生、 需要判斷、容易出錯的 workflow，把它寫成一個 decision-minimal skill。

Hooks and Policies

Hooks / Policies：把希望變成 guardrail

Policies 是「應該怎麼做」；hooks 是「真的不要讓錯的路徑悄悄發生」。 做 agent skills 時，最危險的是只把規則寫進文字，卻沒有執行時的檢查。

Preflight

讀對 context

像 `agent-docs` 這類 preflight 讓 agent 在寫之前拿到 repo 宣告的 required docs 與 validation contract。

Command guard

擋掉危險捷徑

例如 direct `git commit`、direct mutating worktree command、未經 workflow 的 provider 操作。

Finish line

停下前要證明

如果改了 code 卻沒跑 declared validation，finish-line gate 應該擋住草率收尾。

參考： core/hooks、 files-hooks-validation.md、 git-delivery.md

Case Study

forge-cli：為什麼不是直接叫 agent 用 gh / glab

`forge-cli` 展示一個典型 primitive 設計：底層仍然可以使用 `gh` 和 `glab`， 但上層暴露的是 provider-neutral、可 dry-run、可檢查、可被 agent 和 human 使用的 workflow contract。

Repo evidence

來源： sympoies/nils-cli/crates/forge-cli

設計取捨

  • 不隱藏 provider 差異，但把差異包成穩定欄位和 failure mode。
  • 不讓每個 skill 自己拼 provider command。
  • dry-run 先看 mutation plan，避免 agent 直接做不可逆操作。
  • 同一個 command 可以給 agent、shell、Alfred workflow 和 dashboard 使用。
example command 
forge-cli inbox list --item-type pr --format json
forge-cli pr deliver --kind feature --dry-run --format json
forge-cli label audit --catalog manifests/forge-labels.yaml --format json

Same Tool, Many Frontends

同一個工具能力，不同使用場景

這裡的重點是把「工具能力」和「使用入口」分開：CLI 不是只給 agent，也不是只給人。 它是穩定的能力層；Alfred、web dashboard、agent skills 都只是不同場景的入口。

Implementation Loop

如何開始搭建 skills 合集

不要一開始就複製整套 agent-runtime-kit。先用最小閉環建立操作肌肉： 一個 workflow、一個 skill、一個 validation、一個 evidence path。

選 workflow，不選工具

挑一個常做、常卡、常需要判斷的工作，例如 PR delivery、issue triage、release、incident note。

寫 skill contract

先寫何時用、輸入、輸出、failure modes、stop conditions。不要急著把所有 SOP 都放進去。

抽出 CLI 或 script primitive

如果 skill 開始需要 parse provider output、做多步 mutation、產 JSON，就不要再靠 prompt prose，應該抽成 deterministic command。

加 validation 和 hooks

讓重要規則可以被檢查：dry-run、schema、golden、smoke、finish-line gate。hooks 是 guardrail，不是政策替代品。

保留值得保留的學習

用 heuristic system 的精神處理 friction：不是什麼都記，而是把重要或反覆出現的問題變成更好的 contract。
